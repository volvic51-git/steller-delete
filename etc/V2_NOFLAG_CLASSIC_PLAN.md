## 1. 最重要の調査結果：ソルバーは既にクラシックルールで検証している

`js/solver.js` の `isSolvable()`（保証盤面判定・Factory/Hunter/ゲームの共通検証器）は、
**現行ゲームの「地雷除去で数字が減る」ルールを一切シミュレートしていない**。

- `cellAt(r,c).neighborMines` は盤面生成時の**静的な値**を最後まで使う（数字は変化しない）
- 旗（`flagged` Set）は「地雷と確定した」という**知識のマーク**であり、盤面に影響しない
- 勝利判定は「全非地雷セルを開き、全地雷を旗で確定」（solver.js:322-326）

つまり `isSolvable` の中身は**そのまま通常のマインスイーパー（クラシックルール）の
論理充足シミュレーション**である。

### 帰結

| 問い | 答え |
|---|---|
| Factory盤面（18%・72×144）はクラシックルールで成立するか | **成立する（構造的に保証済み）**。Board Hunter / Factory が収集・認定した盤面は全て `isSolvable` を通過しており、それは「クラシックルールで論理のみで解ける」ことの証明そのもの |
| stageEX2（20%・2,074地雷）は | 同上。成功率0.3%をくぐり抜けて収集された盤面は全てクラシック可解 |
| クラシック専用密度・専用盤面は必要か | **可解性の観点では不要**。既存の `data/board/*.json` をそのまま使える |
| 旗禁止モードでは | 旗は情報を生まない（ただのメモ）ので、論理的可解性はクラシックと同一＝**同じく保証済み**。人間がマーク無しで解き切れるかはUX問題（後述） |

現行ルール（地雷除去で数字減算）は情報等価な変形（論証は `V2_RESUME_PERF_PLAN.md §2.1`）
なので、「現行ルールで解ける ⇔ クラシックで解ける」も成り立つ。

## 2. 旗禁止モード（No Flag）仕様案

### A案 / B案の比較と推奨

- **A案**: 非地雷開封率100% → 自動で全地雷を除去 → 既存 `checkWin()`（`removedMines===mineCount`）が発火
- **B案**: 非地雷開封率100% → それ自体をクリア条件にする

**推奨は「B案の判定 ＋ A案の見た目」のハイブリッド。**

- **判定はB案**：`openedNonMine === totalNonMineCells` を新しい勝利条件とする。
  `openCell()` に開封カウンタを1行足すだけで判定でき、O(1)。
  「既存クリア条件とのアンマッチ」は、`checkWin()` の後半（残存セル一括消滅→
  `triggerRescueSequence()`）を共通関数に切り出し、入口の条件だけモード分岐すれば解消する。
- **演出はA案風**：クリア確定後に残存地雷を波状に消滅させる。ただし
  `removeMine()`/`applyMineRemovalEffects()` は**呼ばない**（勝敗確定後は隣接数字の
  再計算が不要）。`isRemoved=true`＋消滅アニメだけを時間差バッチで回す
  「見た目専用の一括消滅」を新設する。

### これで懸念点「地雷2000個の一斉消滅」が解決する

memory [[project-no-flag-mode]] の懸念：2,000個 × `applyMineRemovalEffects` 21.9ms ≒ 44秒。
→ 上記方式なら差分更新そのものをスキップするため、コストは
「メッシュのフェードアウト登録 × 2,000」だけ。既存の `scheduleVanish` の
千鳥バッチ（i*8ms）と同じパターンで演出でき、checkWin の既存一括消滅
（大盤面で実績あり）と同等の負荷に収まる。**実装時に72×144で要実測**だが
原理的な障害はない。

### プレイ中の進捗表示

現行の消滅ベースの演出はほぼ動かない点に注意：
- 旗を立てない＝地雷除去が起きない＝**プレイ中はセルが1個も消滅しない**
- `vanishedNonMine` ベースの RESCUE PROGRESS バー・キャラ透過は 0% のまま
- → **開封率ベース（`openedNonMine`）への切替が必須**（%表示の開封率化は別途相談済み。
  バー・キャラ透過も同カウンタに連動させるのが自然）

### クリアフロー（2026-07-11ユーザー確定）

1. 開封率100% → **自動でクリア成立**（手動トリガーなし。B案採用によりJUDGEアイコンの
   クリックステップは不要になった）
2. SE `EFE_13_judge.mp3` を鳴らし、残存地雷の一括消滅演出を開始（上記バッチ方式。
   「審判が下る」瞬間の音として使用。素材はユーザー準備）
3. → `triggerRescueSequence()`（既存のクリア演出）

**JUDGEアイコン（旧サーチボタン）は転用しない**：現行どおりグレーアウトのまま残す。
用途は検討中（サーチ/ヒント機能の廃止・撤去も保留）。`judgeReveal()`（Factory初手自動開封）
とは別機能なので競合しない。

## 3. クラシックモード（Classic）仕様案

- 旗はトグルのみ（立てる/外す）。`removeMine()` は呼ばない
- 数字セルに旗 → ペナルティなし（誤クリック保険としても機能）
- 数字は生成時の値のまま不変。セルは消滅しない
- 地雷を掘る → ゲームオーバー（現行 `exMode=true` と同じ）
- クリア条件 = **非地雷セル全開封**（No Flag と同じ判定を共用できる。
  旗の有無はクリア判定に含めない：通常のマインスイーパーと同じ）

### 現行実装がすでにクラシック互換な部分

- `digCell` は `hasFlag` セルを掘れない（2261行）＝旗の誤掘削ガードは既にある
- `openCell` のフラッド連鎖は旗セルを飛ばす（2303行）＝本家と同じ挙動
  （誤った旗が連鎖を止めるのも本家仕様どおり。旗を外せば開けられる）

### 注意点

1. **描画負荷**：セルが一切消滅しないため、終盤でも約1万メッシュがフル残存。
   ズームアウト低FPS問題（[[project-perf-zoomout]]、C案未着手）が**全編にわたり顕在化**する。
   72×144をクラシックで遊ぶなら C案（InstancedMesh化）の優先度が上がる。
2. **進捗表示**：No Flag と同じく消滅ベースが機能しない → 開封率ベースへ。
3. **旗の見た目**：現行の旗は立てた瞬間に必ず解決（成功=除去/失敗=GO or ゲージ）される
   前提のため「旗が立ったまま残る」状態の描画・保存は現状ほぼ通らない経路。
   `updateCellVisual` の旗表示自体はあるが、resume/replay で「旗が残った盤面」を
   復元するケースは新規のテスト対象になる。

## 4. 影響範囲整理（依頼のC項）

前提：モードフラグは既存の `exMode`/`loopMode` と同様のパターンで
`gameRule: 'attack' | 'noflag' | 'classic'`（仮）のような1変数に集約するのが良い。
boolean を2つ足すより、排他な3値の方が保存・復元・分岐が単純になる。

| 対象 | 影響 | 内容 |
|---|---|---|
| `flagCell()` (2377) | **中** | noflag: 入口でreturn（＋旗モードボタン/フロートボタン非表示）。classic: `hasFlag` トグルのみの分岐を追加し、`removeMine()`/ゲージ/GO判定をスキップ |
| `removeMine()` (2424) | **小** | 本体は無改修。noflag/classic では呼ばれなくなるだけ。クリア演出用の「見た目専用一括消滅」は別関数として新設 |
| `checkWin()` (2677) | **中** | 勝利条件をモード分岐（attack: `removedMines===mineCount` / noflag・classic: `openedNonMine===totalNonMineCells`）。判定トリガーの追加：`openCell` 側からも勝利チェックを呼ぶ必要がある（現行は地雷除去後にしか呼ばれない）。後半の演出部は共通関数に切り出し |
| `openCell()` (2294) | **小** | `openedNonMine++` を1行追加＋モードによっては勝利チェック呼び出し |
| `saveSuspend()` (3575) | **小** | `meta` に `gameRule` を1フィールド追加。`resumeSuspend()` 側で復元。既存レコードは undefined → 'attack' フォールバック（後方互換） |
| replay | **中** | レコードに `gameRule` を追加（再生時に同ルールで再現）。**`GAME_VERSION`（現"1"）の扱いを決める必要あり**：勝敗判定ロジックが変わるため、厳密には +1 だが、旧リプレイは attack ルール固定で再生すれば互換維持できる（`gameRule` フォールバックで足りる可能性が高い）。classic の「旗が立ったまま」の再生は新規テスト対象 |
| ranking | **中** | `stellarDeleteRanking_stage_N` はステージ単位でルール混在になる。**ルールが違えばタイムは比較不能**なので、キーにルールを含める（例: `..._stage_N_noflag`）か、モード別ステージ（別id）にするかの設計判断が必要。既存記録の移行は不要（attack 扱いのまま） |
| tutorial (manual/) | **小〜中** | シナリオJSON駆動（`manual/scenarios/*.json`）なので、新モード用チュートリアルは**シナリオ追加のみ**で対応可能。既存チュートリアルは「旗＝攻撃」を教えるため、新モードで誤って再生しないよう導線の出し分けだけ必要 |
| Factory盤面 | **なし（重要）** | §1のとおり可解性は保証済み。盤面データ・hash・genVersion・`judgeReveal` 開始モデルすべて無改修で両モードに使える |
| サーチ/ヒント (`refreshSearchAvailability` 等) | **小** | 転用しない（2026-07-11確定）。現行どおりグレーアウトで残す。noflag 中は常時非アクティブになるだけ（`runSolver` の確定セル判定で自然にそうなるか要確認）。用途は検討中 |
| RESCUE PROGRESS / キャラ透過 | **中** | 消滅ベース→開封率ベースへの切替（noflag/classic のみ。attack は現行のまま） |
| debugメニュー | **小** | ルール切替の debug トグルを追加（当面の動作確認入口。2026-07-11にメニュー自体は非表示化済みだが機能は温存されている） |

### 工数感

- **No Flag**：中規模。コア変更（カウンタ・勝利分岐・JUDGE転用・一括消滅演出）は
  局所的だが、保存/リプレイ/ランキングへのフラグ貫通と検証がボリュームの中心。
- **Classic**：No Flag の土台（勝利条件・進捗表示・フラグ貫通）が出来ていれば
  追加分は `flagCell` の分岐と「旗が残る盤面」の保存/復元テストが主。
  **2モード同時に設計して No Flag → Classic の順で実装するのが効率的**。
- 別軸のリスク：Classic は描画性能（C案未着手）への依存が大きい。
  72×144 で快適に遊ばせたいなら C案を先行または並行で。
