# 72×144盤面 フレームレート低下 調査報告

計測日: 2026-07-10 / 対象: stageEX1（72×144・10,368セル・地雷1,866）
方法: `animate()` に一時計測プローブを挿入（計測後に撤去済み・masterに変更なし）。
dev server・デスクトップChrome。進捗状態は resume と同じ instant 再構築で生成。
カメラはズーム7.0（全球が見える距離）固定で3〜4秒×各条件を平均。

---

## 結論（先に要点）

**「進捗とともに重くなる」症状の正体は2つの別問題の重なり**：

1. **【進捗相関・カクつきの正体】旗を立てるたびに発生する巨大ヒッチ**
   `removeMine()` の400ms後処理が**全開封セルの数字テクスチャを毎回作り直す**。
   1セルあたり0.198ms（canvas描画＋CanvasTexture＋Material＋Geometry再生成）×開封数。
   実測: 進捗40%（開封4,559セル）で**旗1本 = 2,834msのフリーズ**＋次フレーム263ms（再アップロード）。
   開封セル数に比例するため「15%で重く感じ、25%超でカクつく」という症状カーブと一致。
2. **【進捗非相関・ベースライン】ズームアウト時のdraw call問題（既知）**
   進捗に関係なく約5,200〜5,800call＝**render 39〜44ms（21〜24fps）**。
   進捗0%でも既に24fps。これは既存の [[project-perf-zoomout]] C案（InstancedMesh化）の領域。

**判定: 完全にCPUバウンド**（毎フレームのcpuSum ≒ フレーム時間。三角形数は15k程度でGPU負荷は誤差）。
内訳は「three.jsのdraw call発行コスト（定常）」＋「数字テクスチャ再生成ストーム（イベント時）」。

## 2. Three.js統計（進捗別）

| 進捗 | calls | triangles | textures | geometries | numberMesh数 | FPS |
|---|---|---|---|---|---|---|
| 0% | 5,210 | 15,330 | 32 | 7,889 | 32 | 24.3 |
| 5% | 5,333 | 15,576 | 387 | 7,976 | 789 | 23.3 |
| 15% | 5,623 | 16,156 | 1,195 | 8,479 | 2,248 | 10.0※ |
| 25% | 5,830 | 16,570 | 1,930 | 9,526 | 3,690 | 3.6※ |
| 40% | 5,305 | 15,556 | 2,247 | 10,062 | 4,559 | 21.5（定常） |

※15%/25%の行は「全数字テクスチャ再生成直後」の計測で、**初回GPUアップロードの
過渡コストを含む**（＝実プレイで旗を立てた直後の状態を代表）。定常なら40%と同傾向で21fps前後。

観測事実:
- **callsは進捗でほぼ増えない**（消滅セルの減少が数字メッシュの増加を相殺）→ 仮説C
  「draw call増加が進捗悪化の原因」は**棄却**（ただしベースライン5,300callが21〜24fpsの原因）。
- **texturesは開封数に比例して単調増加**（0%:32 → 40%:2,247。GPUに載るのは可視化された分のみ。
  数字メッシュ総数は4,559＝各自が個別のCanvasTexture+Material+PlaneGeometryを保有）→ 仮説Bの裏付け。
- geometries 7,889→10,062（+2,173）＝数字メッシュ分の蓄積。

## 3. 切り分け実験

| 実験 | 結果 | 判定 |
|---|---|---|
| 数字メッシュ全非表示（40%） | render 44.3→**27.4ms**・calls 5,323→3,080・FPS 21.5→33.9 | 定常でも数字描画が~17ms/フレームを占有 |
| ズームイン2.2（40%） | **60fps**・calls 147・render 8.6ms | ズームイン時は無問題（視錐台カリングが効く） |
| 中間ズーム4.0（40%） | 21.8fps・calls 5,323 | ズームアウト側で悪化（callsが全量入る） |
| 演出OFF（消滅アニメ即時化＋パーティクル無効、30連続掘削） | 60.0fps → 59.9fps（差なし） | **演出はシロ** |
| **旗1本のヒッチ（40%・ライブ）** | **最大フレーム2,834ms＋263ms**（ベースライン最大61ms） | **カクつきの正体** |

### 旗1本ヒッチの内訳（removeMineの400ms後処理を分解計測）

| 処理 | 時間 |
|---|---|
| **updateCellVisual×4,555開封セル**（createNumberMesh再生成） | **902ms**（0.198ms/セル） |
| （＋次フレームでの可視テクスチャ約2.2千枚の再アップロード） | 約260ms |
| calcNeighbors（全盤面） | 2.0ms |
| checkCascade（全盤面） | 3.2ms |
| vanishNewZeroCells（全盤面） | 0.9ms |

※初回（コールド）実測は2,834ms。分解計測時はdispose/GC分が軽くなり902msだが、
いずれも**開封セル数に比例**する。5%なら~150ms（気づかない）→15%で~450ms+（重い）
→25%で~750ms+（カクつき）→症状の進行と一致。

## 4. 優先順位B: 改善効果予測（計測ベース）

| 施策 | 期待効果 | 根拠 | 規模 |
|---|---|---|---|
| **B案: removeMineの差分更新化**（隣接8セルのみ更新） | **旗ヒッチ 2,834ms → 数ms**（症状に直撃） | ヒッチの99%が全盤面updateCellVisual。差分なら8セル×0.198ms | **小**（計画書 `V2_RESUME_PERF_PLAN.md` §2 設計済み） |
| InstancedMesh化＋数字アトラス（C案） | ズームアウト定常 44ms→数ms（21fps→60fps） | render時間はcalls≒5,300にほぼ比例。数call化で消滅 | 大（別途計画書） |
| NumberTexture/Material共有（数字×色で~30種に集約） | 定常への効果は小。再生成ストームの単価↓・メモリ2,247→~30枚 | 定常44msのうち数字分は~17ms（バインド切替含む）だが、B案導入後は再生成自体が消える | 中 |
| Geometry共有（行単位72種） | フレームへの効果小。メモリ/GC改善のみ | geometryはdraw call数に影響しない | 小 |
| 軽量モード（演出OFF） | **効果ゼロ** | 実測で差なし（60→59.9fps） | — |
| animate内ループ最適化 | 最大1.7ms/フレーム | backCull 1.1+removalAnim 0.55ms | C案に同梱で自然解消 |

### 推奨順序
1. **B案（差分更新）** — 今回の症状（進捗でカクつく）の直接原因を除去。resume高速化と同一施策で、
   計画書に等価性論証・_boardEpochガード設計まで済んでいる。
2. **C案（InstancedMesh＋数字アトラス）** — ズームアウト時の基礎体力（21fps）を60fpsへ。
3. テクスチャ/ジオメトリ共有はC案に同梱（単独ではB案後の残存効果が小さい）。

## 備考
- 計測プローブは撤去済み（masterに変更なし）。再計測したい場合はこの報告書の手順で
  `animate()` にセクション境界の `performance.now()` を再挿入する。
- 15%/25%行の「過渡込み」値は、実プレイの「旗を立てた直後の体感」を代表する点でむしろ実態に近い。
- スマホでは全数値が数倍悪化する想定（同一構図のままスケール）。
